---
description: 구조적 변경(리팩토링)을 동작 변경과 별도로 커밋
---

TDD 원칙에 따라 구조적 변경을 커밋하세요:

1. 모든 테스트가 통과하는지 확인
2. 모든 컴파일러/린터 경고가 해결되었는지 확인
3. 변경사항이 오직 구조적인지 검토:
   - 이름 변경
   - 코드 재구성
   - 추출/인라인화
   - 동작 변경 없음
4. 다음 형식으로 커밋 메시지 작성:
   - "구조: " 또는 "리팩토링: "으로 시작
   - 무엇이 재구성되었는지 설명
   - 예시: "구조: 검증 로직을 메서드로 추출"

기억하세요:
- 테스트가 실패하면 절대 커밋하지 마세요
- 구조적 변경과 동작 변경을 절대 섞지 마세요
- 커밋을 작고 집중적으로 유지
- 커밋 메시지는 구조적 변경임을 명확히 표시해야 함
