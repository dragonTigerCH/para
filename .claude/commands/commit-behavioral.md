---
description: 동작 변경(새 기능 또는 수정)을 구조적 변경과 별도로 커밋
---

TDD 원칙에 따라 동작 변경을 커밋하세요:

1. 모든 테스트가 통과하는지 확인
2. 모든 컴파일러/린터 경고가 해결되었는지 확인
3. 변경사항에 다음이 포함되어 있는지 검토:
   - 동작을 위한 새 테스트
   - 테스트를 통과시키는 구현 코드
   - 구조적 변경만 있는 것이 섞이지 않았는지
4. 다음 형식으로 커밋 메시지 작성:
   - "기능: " 또는 "수정: " 또는 "추가: "로 시작
   - 추가/변경된 동작을 설명
   - 예시: "기능: 이메일 형식 검증 추가"

기억하세요:
- 테스트가 실패하면 절대 커밋하지 마세요
- 구조적 변경과 동작 변경을 절대 섞지 마세요
- 각 커밋은 하나의 논리적 작업 단위를 나타내야 함
- 커밋 메시지는 동작 변경임을 명확히 표시해야 함
